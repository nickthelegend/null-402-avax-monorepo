## Prerequisites

- **Node 22.** The SDK/gateway/MCP packages are exercised on Node 22; if you're
  on a different major version, switch first:
  ```bash
  nvm use 22
  ```
- **Foundry** (`forge`) for the contracts suite.
- Dependencies installed per package (`npm install` inside each package
  directory — they're independent npm packages, not a single workspace).
- The SDK's real-proof test needs the circuit build artifacts. If
  `null-402-circuits/build/` isn't present yet:
  ```bash
  cd null-402-circuits && npm install && npm run build
  ```
- The SDK's `dist/` output is required for the gateway/MCP tests that import
  compiled SDK code:
  ```bash
  cd null-402-sdk && npm install && npm run build
  ```

## null-402-circuits — Circom Groth16 circuit

There's no standalone circuit test script; the circuit's correctness is
proven by the pipeline itself and re-checked by every downstream consumer
(Foundry's real-proof fixture, the SDK's `real-proof` and `evm-live` tests).

```bash
cd null-402-circuits
npm install
npm run build     # compile → powers-of-tau → groth16 setup → witness → prove → verify
```

`build.sh` runs all 6 stages and ends with `snarkjs groth16 verify` against
the freshly generated proof — a non-zero exit means the circuit is broken.
Produces the artifacts everything else consumes: `build/payment_js/payment.wasm`,
`build/payment.zkey`, `build/verification_key.json`.

Coverage: this is the one Circom circuit in the repo (`payment.circom`,
Poseidon Merkle depth 20, 11,491 constraints, 5 public signals
`[nullifier, merkleRoot, payTo, requiredAmount, contextHash]`). It is
chain-agnostic — the same circuit backs both the Foundry verifier and the
SDK's local/EVM verifiers.

### Real-proof test (real snarkjs Groth16)

```bash
npm run test:real
```

Runs `test/real-proof.test.mjs`, which drives an **actual** Groth16 proof
through snarkjs (witness generation + proving + local verification) using
the circuit artifacts from `null-402-circuits/build/`:

- client generates a real Groth16 proof that snarkjs verifies
- `verifyPayment` accepts the real proof (mode=local)
- a tampered proof is rejected by the real verifier
- wrong required amount fails membership/range (the circuit itself refuses
  to produce a witness — this is expected and shows up as a caught
  `Error in template Payment_215` in stdout, not a test failure)

**Result: 4/4 checks passed**, and the runner prints `ALL PASS`.

> **Known issue — benign hang after "ALL PASS".** `test:real` uses snarkjs's
> WASM witness calculator, which spins up worker threads that don't always
> exit cleanly on Node when the process would otherwise finish. The test
> itself completes and reports `4/4 checks passed` / `ALL PASS`; the process
> then hangs instead of exiting. This is why `test:real` is **not** part of
> the default `npm test` — treat the printed `ALL PASS` marker as the actual
> result and kill the process (Ctrl-C, or `timeout`/a watcher that kills on
> the marker) rather than waiting for it to exit on its own.

### Live test (opt-in, real Avalanche Fuji)

```bash
NULL402_RUN_LIVE_TESTS=1 node test/evm-live.test.mjs
# add NULL402_OPERATOR_KEY=0x... to also exercise the settle (on-chain tx) step
```

`test/evm-live.test.mjs` is **skipped by default** (exits 0 immediately)
unless `NULL402_RUN_LIVE_TESTS=1` is set. When enabled it:

1. Calls the real deployed verifier on Fuji via `evmVerifier` (`eth_call`)
   with the same real proof fixture and asserts `valid === true`.
2. Tampers the nullifier and asserts the same verifier call now returns
   `false`.
3. If `NULL402_OPERATOR_KEY` is set, checks whether the fixture's nullifier
   is already spent on-chain (from a prior run) — if so it skips the settle
   step cleanly; otherwise it broadcasts a real `settle()` tx via
   `poolSettle`, then confirms the recipient's nUSD balance increased by the
   expected amount and the nullifier is now marked spent.

This is gated behind an env var (rather than run by default or in CI)
because it makes a real RPC call against Fuji and, with the operator key
set, a **real on-chain transaction** using a funded key that's shared across
this project's agents/demos — running it unconditionally risks nonce
collisions or draining that shared key. **Verified live** during this
documentation pass: the verify + tamper-rejection checks passed against
live Fuji; the settle step was skippable/skipped cleanly per its own logic.

## null-402-gateway — reference edge gateway (Hono)

```bash
cd null-402-gateway
npm install     # pulls the SDK via file:../null-402-sdk — build the SDK first
npm test        # runs test:http && test:http-evm
```

**Result: 7/7** (`test:http` 4 + `test:http-evm` 3).

### `test/http.test.mts` — dev mode (4 tests)

Runs the real Hono app in-process (`app.fetch()`) with `VERIFY_MODE="dev"`
and an in-memory KV stand-in — no wrangler/workerd, no chain calls:

- `GET /health` → 200, dev mode reported
- `GET /v1/price/BTC` with no payment → 402 + payment terms
- `GET /v1/price/BTC` with a valid (dev-scaffold) proof → 200 + data +
  `X-Privacy` header
- replay of the same proof → 402

`VERIFY_MODE=dev` is an insecure local scaffold (`DEV_SHARED_SECRET`, no
chain interaction at all) — useful for exercising the HTTP/402 state machine
without touching a network, never for production.

### `test/http-evm.test.mts` — EVM mode against a **local stub RPC** (3 tests)

Runs the same app with `VERIFY_MODE="evm"`, but points `EVM_RPC_URL` at a
**local, in-process JSON-RPC stub server** (`http://127.0.0.1:<port>`), not
live Avalanche Fuji:

- `VERIFY_MODE=evm` + stub `eth_call` returns `true` → 200 (real gateway
  code path: viem `createPublicClient` + `readContract` + `eth_call`)
- `VERIFY_MODE=evm` + stub `eth_call` returns `false` → 402 payment-rejected
- `VERIFY_MODE=evm` + `KNOWN_ROOTS` set rejects an unlisted root as
  unknown-root (400)

This proves the gateway correctly wires an `eth_call` verify result into
its 402/200/400 decision using production code paths, entirely offline. It
does **not** re-test Groth16 pairing math on-chain — that's covered by
`null-402-contracts/test/Null402.t.sol` (Foundry, real proof, real Fuji-style
EVM execution) and `null-402-sdk/test/real-proof.test.mjs` (real snarkjs).

**Dev vs EVM mode, summarized:**

| `VERIFY_MODE` | Verifier | Chain calls | Use |
|---|---|---|---|
| `dev` | shared-secret string match | none | local iteration on the HTTP layer only |
| `evm` | `evmVerifier` → `eth_call` to the deployed Solidity verifier | yes — real `eth_call` (RPC endpoint is just a config string, so tests point it at a local stub instead of Fuji) | the real, production verification path |

## null-402-mcp — MCP server tool handlers

```bash
cd null-402-mcp
npm install
npm test
```

**Result: 20/20** (`node test/lib.test.mjs`).

All four MCP tools (`create_wallet`, `wallet_status`, `deposit`, `pay`) are
unit-tested via dependency injection — the handlers take mocked viem
`publicClient`/`walletClient` objects and a temp wallet directory, so no
real network, gas, or funded key is touched:

- `create_wallet`: generates + persists to the injected path, behavior with
  and without a funder key, funder-failure resilience, idempotency on a
  second call
- `wallet_status`: reports "no wallet" without touching the network, reads
  balances via the mocked client, degrades to `?` instead of throwing on a
  balance-read failure
- `deposit`: no-wallet guard, mints the note denom + calls `poolDeposit` with
  the right args + persists the note, propagates mint/escrow failures without
  writing a note
- `pay`: no-wallet / no-notes guards, endpoints that don't require payment
  short-circuit, missing payment terms rejected, the full 402 → prove →
  retry-with-`X-PAYMENT` happy path (marks the note spent), a non-200
  settlement response leaves the note unspent, an unreachable URL surfaces
  as a thrown error rather than a crash
- one sanity check that the module-level config defaults (used by
  `index.mjs`/`selftest.mjs`) are well-formed

Not exercised by `npm test` (require real funded keys / a live gateway, so
they're manual/demo scripts, not part of the automated suite):
`npm run selftest` (real local gateway + deployed testnet contracts) and
`npm run agent` / `npm run agent:groq` (autonomous agent demos against live
Fuji).

## Live vs local, at a glance

| Suite | Network | Real ZK proving | Real on-chain tx |
|---|---|---|---|
| contracts (`forge test`) | local EVM (Foundry) | yes (fixture) | no (local chain, not Fuji) |
| sdk `npm test` | none | no (dev prover) | no |
| sdk `npm run test:real` | none | **yes** (snarkjs) | no |
| sdk `npm run test:live` (`NULL402_RUN_LIVE_TESTS=1`) | **live Fuji** | no (uses fixture proof) | only if `NULL402_OPERATOR_KEY` set, and only if not already spent |
| gateway `test:http` | none | no | no |
| gateway `test:http-evm` | local stub RPC | no | no |
| mcp `npm test` | none (mocked clients) | no | no |

Everything except the SDK's opt-in live test runs fully offline/local and is
safe to run repeatedly and in CI. The live test is intentionally excluded
from any default/CI run and must be invoked explicitly with
`NULL402_RUN_LIVE_TESTS=1` to protect the shared funded operator key from
concurrent/duplicate broadcasts.
